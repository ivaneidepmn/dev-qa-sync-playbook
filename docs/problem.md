# The Problem

In many software teams a single QA engineer is responsible for supporting multiple development squads.

While this model can work, it often creates operational challenges:

- QA becoming a bottleneck
- Developers lacking context when reporting issues
- Inconsistent bug reports
- Testing cycles delayed near release time

These issues usually do not come from individuals but from **process gaps between development and quality workflows**.

---

## Common Friction Points

1. QA involvement too late in development
2. Lack of clear acceptance criteria
3. Bug reports without reproducible steps
4. Developers relying entirely on QA for validation
5. QA overwhelmed by multiple teams

---

## Goal of This Study

Understand these friction points and propose a **structured collaboration model between Dev and QA**.
